# ops/handoff — Agent 任务交接协议

> 本地 Agent「指挥」服务器侧 Agent 的最简信道：**Git 当邮箱**。
> 不依赖 SSH、不常驻服务、不开新端口、不加攻击面；天然多机可见、有完整历史。

## 目录

```
ops/handoff/
├── README.md      ← 本文件(协议)
├── inbox/         ← 待办:派发方写任务文件到这里
└── done/          ← 已完成:执行方做完后把文件移到这里并回写结论
```

## 流程

1. **派发**(通常是本地 Coco / 你)：在 `inbox/` 新建一个任务文件
   `YYYYMMDD-HHmm-<短标题>.md`(用下方模板)，`git commit && git push`。
2. **认领 + 执行**(服务器侧 Agent / Codex)：`git pull`，挑 `inbox/` 里 `状态: 待认领`
   的文件，改 `状态: 进行中` 并填 `执行: <你的名字>`，commit。做完把文件
   **`git mv` 到 `done/`**，在文件尾部「结论」段回写结果/产物路径/验证命令，commit。
3. **验收**：派发方 `pull` 看 `done/`，不满意就在 `inbox/` 再开一条引用它。

> 一次只认领一个文件、改完即 commit，能最大程度避免多 worktree 撞同一文件。
> 真冲突时谁先 push 谁赢，后者 `pull --rebase` 重挑。

## 任务文件模板

```markdown
---
状态: 待认领          # 待认领 | 进行中 | 已完成 | 已取消
派发: coco
执行:                 # 认领时填
优先级: 高            # 高 | 中 | 低
创建: 2026-06-23 14:30
---

## 目标
<一句话说清要什么>

## 上下文
<相关文件/端点/背景;尽量给路径,少让执行方猜>

## 验收标准
- [ ] <可验证的条件 1>
- [ ] <可验证的条件 2>

## 结论            # 执行方完成后回写
<做了什么 / 产物在哪 / 怎么验证 / 遗留问题>
```

## 约定

- 文件名时间戳保证排序与唯一;标题用 kebab 或中文短词。
- 只在 `inbox/` ↔ `done/` 之间移动文件,不要删除(留痕)。
- 任务涉及改代码时,执行方在独立 `git worktree` 里做,结论里写清分支/commit。
- 与人看的项目看板(`ops/board/`)分工:**handoff = Agent 间一次性任务**,
  **board = 人看的进度/需求/里程碑**。
