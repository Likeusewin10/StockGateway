# 远程调用手册

供 Mac / 其他项目 / MCP 客户端跨网络调用本股票数据服务。本手册合并了**两条独立接入通道**,二者共用同一台服务器、同一个 `API_KEY`、同一个 ngrok 会话:

| 通道 | 公网地址 → 端口 | 适用对象 | 能力 |
|---|---|---|---|
| **① REST + WebSocket** | `https://jtx.ngrok.app` → 8000 | 任意语言/项目,发 HTTP 即可(**不装任何 SDK**) | EM/iFinD/Wind/TDX 四源取数、通用透传、交易端点、WS 实时推送 |
| **② MCP 网关** | `https://jtx-mcp.ngrok.app/mcp` → 8765 | MCP 客户端(Claude Code / Cursor / 桌面版) | **312 个工具**:4 Agent(在服务器起 claude/codex 干活)+ 32 iFinD + 11 妙想(东财)+ 258 Tushare |

```
                                   ┌─ :8000 REST/WS  (jtx.ngrok.app)     ── EM/iFinD/Wind/TDX 取数+交易+推送
本机 ──X-API-Key──> ngrok(一个会话)─┤
                                   └─ :8765 MCP 网关 (jtx-mcp.ngrok.app) ─┬─ agent_*      : 起 claude/codex 子进程
                                                                          ├─ ifind_*      : 转发同花顺上游取数
                                                                          ├─ mx_ds_*      : 转发妙想(东财)上游取数
                                                                          └─ tushare_ds_* : 转发 Tushare 上游取数
```

> **怎么选通道**:只想在自己代码里取数/下单 → 用**通道 ①**(HTTP,通用)。用 Claude Code / Cursor 这类 MCP 客户端、想让 AI 直接调数据或指挥服务器端 Agent → 用**通道 ②**(MCP)。两者可同时用,钥匙同一把。

## 二、通道 ①:REST + WebSocket 取数(`jtx.ngrok.app`)

调用方**不需要安装任何股票 SDK、也不需要装任何客户端**,只要发 HTTP GET/POST 请求 + 带密钥鉴权头即可。

### 2.1 接口一览

所有便捷端点均为 **GET**,参数走查询字符串,返回 JSON。

| 接口 | 说明 | 必填参数 | 可选参数 |
|---|---|---|---|
| `/health` | 健康检查(免鉴权) | 无 | 无 |
| `/em/csd` | 东财序列数据 | `codes` `indicators` `startdate` `enddate` | `options` |
| `/em/css` | 东财截面数据 | `codes` `indicators` | `options`(如 `TradeDate=20240105`) |
| `/ths/history` | 同花顺历史行情 | `codes` `indicators` `begin` `end` | `params` |
| `/ths/basic` | 同花顺基础/截面 | `codes` `indicators` | `params` |
| `/ths/realtime` | 同花顺实时行情 | `codes` `indicators` | `params` |
| `/wind/wsd` | 万得日序列数据 | `codes` `fields` `startdate` `enddate` | `options` |
| `/wind/wss` | 万得日截面数据 | `codes` `fields` | `options`(如 `tradeDate=20260629`) |
| `/tdx/bars` | 通达信 K 线 | `codes` | `fields` `period` `start` `end` `count` `dividend` |
| `/tdx/snapshot` | 通达信行情快照(含五档) | `code` | `fields` |
| `/tdx/stock_info` | 通达信基础财务/资料 | `code` `fields` | 无 |
| `/tdx/financial` | 通达信专业财务(Fn 字段) | `codes` `fields` | `start` `end` `report_type` |
| `/tdx/sector` | 通达信板块成分股 | `block` | `block_type` `list_type` |
| `/tdx/stock_list` | 通达信按市场取代码 | `market` | `list_type` |
| `/tdx/trading_dates` | 通达信交易日历 | `market` | `start` `end` `count` |

参数约定(**四源各有差异,注意别混用**):
- **EM(东财)**:`codes` / `indicators` 多个用**逗号**分隔,日期格式 `YYYY-MM-DD`。
- **iFinD(同花顺)**:`codes` 多个用**逗号**,`indicators` 多个用**分号**,日期格式 `YYYY-MM-DD`。
- **Wind(万得)**:`codes` / `fields` 多个均用**逗号**分隔,日期格式 `YYYY-MM-DD`;`options` 为 SDK 原生 options 串(如 `tradeDate=20260629`、`PriceAdj=F`)。**取数免开关;交易/写类(`torder` 等)默认 503,需服务端开 `WIND_TRADING_ENABLED`**。
- **TDX(通达信)**:`codes` / `fields` 多个均用**逗号**分隔;证券代码须为**标准格式**(6 位 + 市场后缀,如 `600519.SH`/`000001.SZ`);日期格式 `YYYYMMDD`(注意:与前三源的 `YYYY-MM-DD` 不同)。`period` 取 `1d/1w/1m/5m/15m/30m/60m`,`dividend` 取 `none/front/back`。`market` 为市场类型码(`5` 所有 A 股、`50` 沪深 A 股、`23` 沪深 300、`52` 科创板、`53` 北交所等,全集见 `/tdx/methods` 提示与官方文档)。**取数只读;交易走带护栏的 `/tdx/order`|`/tdx/cancel`(默认关+dry-run),透传 `/tdx/call` 仍拦交易/账户/写/会话类 403**。依赖本机通达信终端已开启并登录。

### 2.2 通用透传接口(覆盖四个 SDK 的全部方法)

上面这些是常用便捷封装;**其余所有 SDK 方法**(EDB 宏观、专题报表 `ctr`、条件选股 `cps`、板块 `sector`、交易日历、智能问财 `iwencai`、数据池 `DataPool`、财报 `ReportQuery`、通达信分红送配/可转债/新股等)无需逐个加端点,统一走**通用透传**接口,用 **POST** 传参:

| 接口 | 方法 | 说明 |
|---|---|---|
| `GET /em/methods` | GET | 列出东财 `c` 的所有方法及类别标注(均可调用,状态以 `ok` 开头) |
| `POST /em/call/{method}` | POST | 调用 `c.<method>(*args)`,body:`{"args":[...], "options_pandas":true}` |
| `GET /ths/funcs` | GET | 列出 iFinD 所有 `THS_` 函数及类别标注(均可调用,状态以 `ok` 开头) |
| `POST /ths/call/{func}` | POST | 调用 `THS_<func>(*args, **kwargs)`,body:`{"args":[...], "kwargs":{...}}` |
| `GET /wind/methods` | GET | 列出万得 `w` 的所有方法及类别标注(交易类标 `blocked`,调用 403) |
| `POST /wind/call/{method}` | POST | 调用 `w.<method>(*args, usedf=...)`,body:`{"args":[...], "usedf":true}` |
| `GET /tdx/methods` | GET | 列出通达信 `tq` 的所有方法及类别标注(交易/账户/写/会话类标 `blocked`,透传 403;交易走 `/tdx/order`|`/tdx/cancel`) |
| `POST /tdx/call/{method}` | POST | 调用 `tq.<method>(*args, **kwargs)`,body:`{"args":[...], "kwargs":{...}}` |

- `options_pandas`(仅 EM)默认 `true`:自动给最后一个 options 串补 `Ispandas=1`,返回转成 JSON 记录数组。
- **全部方法均可调用,不再拦截**。`/em/methods`、`/ths/funcs` 会给出类别标注(均以 `ok` 开头),方便你判断用法:
  - `ok`:常规取数。
  - `ok(交易/组合)`:东财组合管理类(`pcreate/porder/pquery/preport/pctransfer/pdelete`),操作虚拟组合持仓,无真实资金流出。
  - `ok(异步...)`:异步推送类(EM `csq/chq/cst/cnq`、iFinD `THS_Asy*`)。回调模型不适配同步 HTTP,**请改用 WebSocket**(见 2.5),直接同步调用拿不到推送结果。
  - `ok(会话管理,谨慎...)`:⚠️ 会改服务自身的 SDK 会话状态,**会影响整个服务的所有调用方**。例如 `POST /em/call/stop` 会登出整个服务的东财会话、`THS_iFinDLogout` 会登出同花顺会话,导致其他人取数失败直到服务重启自愈。除非你清楚后果,否则不要调用这类方法。
- 先调 `/em/methods`、`/ths/funcs` 看清方法名怎么拼、属于哪类。

```bash
# 东财 EDB 宏观数据(原来没有专门端点,走透传)
curl -X POST -H "X-API-Key: <API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"args":["EMM00087117","IsPublishDate=1,RowIndex=1"]}' \
  "https://jtx.ngrok.app/em/call/edb"

# 东财组合下单(虚拟组合持仓加减,无真实资金):porder(combincode, orderdict, remark)
# orderdict: code/volume/price/date 列表对齐;optype 1买入 2卖出
curl -X POST -H "X-API-Key: <API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"args":["mypf.PF",{"code":["300059.SZ"],"volume":[1000],"price":[13.11],"date":["2024-01-05"],"optype":[1]},"备注"],"options_pandas":false}' \
  "https://jtx.ngrok.app/em/call/porder"

# 同花顺 EDB(位置参数顺序同 SDK:indicators, param, begin, end)
curl -X POST -H "X-API-Key: <API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"args":["M001620326","","2024-01-01","2024-06-01"]}' \
  "https://jtx.ngrok.app/ths/call/EDB"

# 万得板块成分 wset(args:功能名 + options 串;usedf 默认 true 转 JSON 记录)
curl -X POST -H "X-API-Key: <API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"args":["sectorconstituent","date=2026-06-29;windcode=000300.SH"]}' \
  "https://jtx.ngrok.app/wind/call/wset"

# 万得宏观 edb(args:指标ID, 起, 止)。交易类(torder/tlogon 等)默认 503,需服务端开 WIND_TRADING_ENABLED
curl -X POST -H "X-API-Key: <API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"args":["M0001385","2026-01-01","2026-06-01"]}' \
  "https://jtx.ngrok.app/wind/call/edb"

# 通达信分红送配 get_divid_factors(args:证券代码, 起, 止)。
# 交易类(order_stock/cancel_order_stock/query_* 等)与写客户端类(send_*)一律 403
curl -X POST -H "X-API-Key: <API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"args":["688318.SH","",""]}' \
  "https://jtx.ngrok.app/tdx/call/get_divid_factors"
```

### 2.3 各接口示例

下列 curl 可直接复制运行。浏览器无法带请求头,仅适合 `/health` 或临时调试。

#### /em/csd —— 东财序列数据
某只票一段时间的指标序列。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/em/csd?codes=300059.SZ&indicators=CLOSE&startdate=2024-01-02&enddate=2024-01-05"
```

#### /em/css —— 东财截面数据
多只票在某一天的指标快照。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/em/css?codes=300059.SZ,000002.SZ&indicators=OPEN,CLOSE&options=TradeDate=20240105"
```

#### /ths/history —— 同花顺历史行情
注意 `indicators` 多个用**分号**。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/ths/history?codes=300033.SZ&indicators=open;close&begin=2024-01-02&end=2024-01-05"
```

#### /ths/basic —— 同花顺基础/截面数据
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/ths/basic?codes=300033.SZ&indicators=ths_stock_short_name_stock"
```

#### /ths/realtime —— 同花顺实时行情
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/ths/realtime?codes=300033.SZ&indicators=latest"
```

#### /wind/wss —— 万得截面数据
多只票在某一天的指标快照;`codes` / `fields` 均用逗号分隔。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/wind/wss?codes=600519.SH,000001.SZ&fields=sec_name,close,pe_ttm&options=tradeDate=20260629"
# [{"index":"600519.SH","SEC_NAME":"贵州茅台","CLOSE":1194.96,"PE_TTM":18.06}, ...]
```

#### /wind/wsd —— 万得日序列数据
某只票一段时间的指标序列。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/wind/wsd?codes=600519.SH&fields=close,volume&startdate=2026-06-23&enddate=2026-06-29"
# [{"index":"2026-06-23","CLOSE":1222.45,"VOLUME":5800405.0}, ...]
```

#### /tdx/bars —— 通达信 K 线
多只票的 K 线,返回 `{字段: 记录数组}`,每个字段一个 DataFrame(index 为日期)。日期格式 `YYYYMMDD`。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/tdx/bars?codes=600519.SH,000001.SZ&period=1d&count=3&dividend=front"
# {"Close":[{"index":"2026-06-30T00:00:00","600519.SH":1185.49,"000001.SZ":...}], "Open":[...], ...}
```

#### /tdx/snapshot —— 通达信行情快照
单只证券最新行情,含五档买卖盘、内外盘、均价等。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/tdx/snapshot?code=600519.SH"
# {"Now":"...","LastClose":"...","Buyp":["..."],"Sellp":["..."],"ErrorId":"0", ...}
```

#### /tdx/stock_info —— 通达信基础财务/资料
`fields` 必填(逗号分隔)。免下载专业财务数据即可取。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/tdx/stock_info?code=600519.SH&fields=Name,BelongHS300,BelongRZRQ"
# {"Name":"贵州茅台","BelongHS300":"1","BelongRZRQ":"1","ErrorId":"0"}
```

#### /tdx/financial —— 通达信专业财务
`fields` 为 `Fn` 字段(需客户端先下载专业财务数据)。`report_type`:`report_time` 按截止日 / `announce_time` 按公告日。
```bash
curl -H "X-API-Key: <API_KEY>" \
  "https://jtx.ngrok.app/tdx/financial?codes=688318.SH&fields=Fn193,Fn194,Fn195&report_type=announce_time"
```

#### /tdx/sector、/tdx/stock_list、/tdx/trading_dates —— 板块/代码/交易日
```bash
# 板块成分股(block_type=1 表示传自定义板块简称;list_type=1 返回代码+名称)
curl -H "X-API-Key: <API_KEY>" "https://jtx.ngrok.app/tdx/sector?block=钛金属&block_type=1&list_type=1"
# 按市场取代码(market=23 沪深300;list_type=1 带名称)
curl -H "X-API-Key: <API_KEY>" "https://jtx.ngrok.app/tdx/stock_list?market=23&list_type=1"
# 交易日历(需客户端先下载上证指数 999999 盘后数据,仅 A 股)
curl -H "X-API-Key: <API_KEY>" "https://jtx.ngrok.app/tdx/trading_dates?market=SH&start=20260101&count=10"
```

> **⚠️ TDX 数据预下载前提(部分端点依赖本机通达信客户端先下好数据)**:
> - `/tdx/bars`(日线/分钟线)、`/tdx/trading_dates`:需在客户端「**系统→盘后数据下载→沪深京日线**」勾「日线和实时行情数据」下完整历史(含上证指数 999999)。**已下载并真机验证可用**。分钟线另在「沪深京分钟线」标签页下载。
> - `/tdx/financial`(`Fn` 专业财务字段):需在客户端单独下载「**专业财务数据包**」(不在「盘后数据下载」对话框内),否则 Fn 字段返回 `null`。只要基础财务/资料请改用 `/tdx/stock_info`(免下载,已验证有值)。
> - `/tdx/snapshot`、`/tdx/stock_info`、`/tdx/stock_list`、`/tdx/sector`:实时拉取,无需预下载。

### 2.4 交易端点(TDX / Wind,受控放开)

> 🔴 **交易默认关**:TDX 写操作需服务端 `.env` 设 `TDX_TRADING_ENABLED=true` 才放行(否则 `503`);且下单**默认 dry-run**,`confirm=true` 才真发。真发前过四道护栏(金额/白名单/当日笔数/交易时段)+ 全量审计。Wind 交易类透传另受 `WIND_TRADING_ENABLED` 控制。**公网放开真实交易前请与服务方确认。**
>
> 🔴 **TDX 交易端点额外依赖「券商交易账户已登录」**:通达信终端的**行情登录**(看行情、取数)与**交易登录**(连券商资金账户)是两回事。取数端点(`/tdx/bars` 等)只要行情登录即可;交易类端点还需在终端「交易」板块登入券商资金账户,且服务端 `.env` 配好 `TDX_ACCOUNT_ID`(券商资金账号)。未就绪时 `/tdx/health` 返回 `{"connected":false,...}`,`/tdx/asset` 等返回 `502`;此时**取数不受影响**。**切换终端登录账号后须重启服务**(服务缓存账户句柄,不会自动刷新),可用 `/tdx/health` 自检恢复情况。

```bash
# 交易账户自检(connected=true 才能用下面的资产/持仓/委托/下单)
curl -H "X-API-Key: <API_KEY>" "https://jtx.ngrok.app/tdx/health"

# 查询账户资产 / 持仓 / 当日委托(读端点,不受交易开关限制)
curl -H "X-API-Key: <API_KEY>" "https://jtx.ngrok.app/tdx/asset"
curl -H "X-API-Key: <API_KEY>" "https://jtx.ngrok.app/tdx/positions"
curl -H "X-API-Key: <API_KEY>" "https://jtx.ngrok.app/tdx/orders?cancelable=1"

# 下单(dry-run 预览,不带 confirm):side=buy|sell,price_type=limit|market(限价必带正 price)
curl -X POST -H "X-API-Key: <API_KEY>" -H "Content-Type: application/json" \
  -d '{"side":"buy","stock_code":"600519.SH","volume":100,"price_type":"limit","price":1500.0}' \
  "https://jtx.ngrok.app/tdx/order"
```

### 2.5 WebSocket 实时推送(真·订阅推送)

需要持续盯盘、不想轮询时,用 WebSocket 长连接。SDK 行情回调一触发就顺着连接推下来。

| 端点 | 说明 |
|---|---|
| `WS /em/ws?key=<API_KEY>` | 东财实时行情推送(底层 `csq`) |
| `WS /ths/ws?key=<API_KEY>` | 同花顺实时行情推送(底层 `QuotesPushing`) |

- **鉴权走 query 参数** `?key=`(WebSocket 不便带请求头),值同 `API_KEY`;错误则返回 error 帧并断开。
- 连接后发 JSON 文本帧订阅,可多次订阅多个标的;**断开连接时服务端自动退订全部**。
- 协议(客户端 → 服务端):
  - 订阅:`{"action":"subscribe","codes":"300059.SZ","indicators":"now,open"}`
  - 退订:EM `{"action":"unsubscribe","serial":<订阅号>}`;iFinD `{"action":"unsubscribe","codes":"300033.SZ","indicators":"latest"}`
- 服务端 → 客户端帧:`{"event":"subscribed",...}` / `{"event":"quote","data":...}` / `{"event":"error","msg":...}`
- ⚠️ **依赖账号实时推送权限**:当前东财账号 `csq` 无权限(订阅返回 `10001012 insufficient user access`);同花顺可订阅成功。**非交易时段无 tick 推送属正常**。
- ⚠️ 公网经 ngrok 的长连接偶发断开,客户端需自带重连逻辑。

```python
# pip install websockets
import asyncio, json, websockets

async def main():
    url = "wss://jtx.ngrok.app/ths/ws?key=<API_KEY>"
    async with websockets.connect(url) as ws:
        await ws.send(json.dumps({"action":"subscribe","codes":"300033.SZ","indicators":"latest"}))
        async for msg in ws:
            print(json.loads(msg))

asyncio.run(main())
```

## 三、通道 ②:MCP 网关(`jtx-mcp.ngrok.app`)

本机(如 Mac)通过 MCP 协议,经公网调用这台 Windows 服务器上的 **312 个工具**:4 个 Agent 工具(在服务器上起 claude/codex 干活)+ 32 个 iFinD 股票数据工具 + 11 个妙想(东方财富)数据工具 + 258 个 Tushare 数据工具。

> **主用 Agent = 服务器端的 Claude Code 子进程(`engine=claude`)。** 这是你日常指挥、主要沟通的对象;`codex` 仅作备选引擎。
> 子进程是现起、cwd 锁死服务器 StockSDK 目录、带服务器 `CLAUDE.md` 身份的一次性进程;但**支持上下文接力**——首轮返回 `session_id`,后续把它传回即续接全部历史(见 3.2)。客户端只保管一个几十字节的 `session_id` 字符串,历史永留服务器。

### 3.1 客户端接入(如 Mac)

### 3.2 指挥服务器端 Agent 干活(异步三步 + 上下文接力)

工具:`agent_agent_run` / `agent_agent_status` / `agent_agent_result` / `agent_agent_sessions`。
异步模型(避开 MCP 同步超时):起任务拿 `task_id` → 轮询 status 至终态 → 取 result。

> 直接用自然语言让客户端 Claude 代劳即可,例:
> 「用 stocksdk-gateway 的 agent_agent_run,engine=claude,prompt="列出当前目录前3个文件",轮询到 done 再取结果」

- `engine`:限 `claude` / `codex`。**主用 `claude`**(服务器端 Claude Code 子进程,默认沟通对象);`codex` 备选。
- 子进程**全自动非交互**,cwd 锁死服务器 StockSDK 目录;服务器 `CLAUDE.md` 随 prompt 自动加载。
- status:`running` / `done` / `failed` / `timeout` / `unknown`。
- 🔴 全自动 = 拿到 `API_KEY` 即可让服务器无确认执行任意命令。**key 只发可信机器。**

**会话语义:支持上下文接力(session_id 回传 / 续传)** — `agent_agent_run` 接受 `engine` + `prompt` + 可选 `session_id` 三个字段。会话历史**永远留在服务器**,客户端只需保管一个几十字节的 `session_id`:

```
第1轮  agent_run(engine=claude, prompt="记住数字 42")                  → 返回里带 session_id S
第2轮  agent_run(engine=claude, prompt="我让你记的数字是几?", session_id=S) → 自动加载历史,答 42
第3轮  agent_run(engine=claude, prompt="再加 1 呢?", session_id=S)         → 继续累积
```

- **不传 `session_id`** = 全新会话(向后兼容)。**传入 `session_id`** = 续接该会话全部历史(须为合法 UUID,非法直接 `failed`)。
- `task_id` 仍只是这一次任务的轮询句柄(给 status/result 用),**与 `session_id` 是两码事**——`session_id` 才是跨轮续接的钥匙。
- **两引擎拿 id 的时机不同**:`claude` 服务器**预生成** session_id,**首轮 `agent_run` 即时返回里就带**;`codex` 的 id 由 codex 自己生成,**须轮询到终态后**从 `agent_status`/`agent_result` 里拿。
- **看历史会话标题列表(仿原生 `/resume`)**:调 `agent_agent_sessions`(可选 `limit`)列出本项目下 claude+codex 的历史会话,每条含 `session_id` + 派生标题 + 时间,按修改时间倒序。挑一条的 `session_id` 直接喂给 `agent_run` 即续接。
- 会话历史存在 claude/codex 各自的磁盘存储,与网关任务表生命周期**解耦**——任务表淘汰不丢会话。续接时若服务器侧对应会话历史已被 CLI 清理,子进程会非零退出走 `failed`,据 error 重开首轮即可。

## 四、返回与错误码(通道 ① REST)

| 状态码 | 含义 | 处理 |
|---|---|---|
| `200` | 成功,body 为 JSON 数据 | 正常解析 |
| `400` | 参数与 SDK 方法签名不匹配(透传时常见) | 对照 `/em/methods`、`/ths/funcs` 和 SDK 文档核对 args |
| `401` | 缺少或错误的 `X-API-Key` | 检查请求头是否带、key 是否与服务器 `.env` 一致 |
| `403` | 调用了私有方法(名字以 `_` 开头),或 TDX 透传里的交易/账户/写/会话类方法 | 用专用端点(`/tdx/order`\|`/tdx/cancel`\|`/tdx/asset` 等)或只调公开方法 |
| `404` | 方法名拼错 / 不存在 | 用 `/em/methods`、`/ths/funcs`、`/wind/methods`、`/tdx/methods` 确认方法名 |
| `429` | 触发限流(每 IP 每 60 秒最多 60 次取数请求) | 降低请求频率,稍后重试;高频批量取数请合并参数 |
| `502` | SDK 登录失败 / 取数无权限 / 流量用尽 / 终端未开;TDX 交易端点另指**券商交易账户未登录**(`stock_account` 返回 -1,取数不受影响) | 看 body 里的原始错误码,多为账号或本机终端侧问题;TDX 交易类先查 `/tdx/health` |
| `503` | 交易未启用:QMT/TDX 的 `/…/order`\|`/…/cancel` 或 Wind 交易类透传,需服务端开对应 `*_TRADING_ENABLED` | 联系服务方开启交易总开关 |
| 连接超时 | 服务/隧道没起 或 网络不通 | 见第五节排查 |

> **限流**:服务对取数接口(`/em/*`、`/ths/*`、`/wind/*`、`/tdx/*`,含透传)按来源 IP 限流,**每 60 秒最多 60 次**,超出返回 `429`。`/health` 不限流。限流计数在服务进程内,重启即重置。
> **CORS(浏览器跨域)**:默认**拒绝**所有跨域请求。如需从浏览器前端直连,需在服务器 `.env` 的 `CORS_ORIGINS` 把你的站点域名加入白名单(逗号分隔);服务端调用(curl/requests/后端)不受此限制。

## 五、排错

### 通道 ① REST/WS 连不上(按顺序)

1. **先探活**:`curl https://jtx.ngrok.app/health`。通 → 服务正常,问题在你的请求(多半是 key)。不通 → 往下查。
2. **瞬时断连**:公网节点偶发抖动,先原样重试 1~2 次。
3. **服务层**(需到服务器上):两套计划任务托管,均登录自启 + 每分钟自愈:
   - 数据服务 `StockDataService`:`schtasks /Run /TN StockDataService`;状态 `Get-ScheduledTask -TaskName StockDataService | Get-ScheduledTaskInfo`。
   - 隧道 `NgrokTunnel`:`schtasks /Run /TN NgrokTunnel`;日志在项目目录 `ngrok.log`。

### 通道 ② MCP 网关

| 现象 | 排查 |
|---|---|
| 客户端连不上 / `/mcp` 失败 | **先在客户端机器上一步定位**:`curl -s -o /dev/null -w "%{http_code}" -X POST https://jtx-mcp.ngrok.app/mcp`。**三态区分谁的锅**:得 **`401`** = 隧道+服务端全通,问题在客户端 MCP 配置(URL 漏 `/mcp`、key 没填/填错);得 **`502`** = 隧道在线但**服务端网关进程死了**(见下条 502);得 **`000`/超时** = 客户端到 ngrok 出网不通(见代理/DNS 条)。仍存疑再上服务器查:① 本机 `curl http://127.0.0.1:8765/mcp`(网关挂了→拉起 `MCPGatewayBoot` 或 `start_mcp_gateway.bat`);② 公网隧道(ngrok 掉了→重启计划任务 `NgrokTunnel`) |
| **公网 `502`:隧道通但网关后端死** | 🔴 客户端(或服务端自测)打 `https://jtx-mcp.ngrok.app/mcp` 得 **`502`** = **ngrok 隧道在线、但转发目标 `localhost:8765` 没进程**(网关挂了)。这是**服务端**问题,加不加代理都是 502。判据:服务器上 `netstat -ano \| findstr :8765` 无 `LISTENING`;`curl http://127.0.0.1:8765/mcp` 得 `000`;而 REST(8000)`/health` 仍 `200`。修法:拉起网关看门狗 `schtasks /Run /TN MCPGatewayBoot`(或 `start_mcp_gateway.bat`),约 12s 后 8765 重新监听,公网 502→401。**历史根因**:`MCPGatewayBoot` 计划任务曾缺"登录启动 + 每分钟重复"触发器(不同于 `StockDataService`/`NgrokTunnel`),承载它的 cmd 窗口一旦整个消失就不再自愈;已改为同款双触发器根治。 |
| **客户端 `000`:ngrok 域名需走代理** | 🔴 **先排除服务端 502(上一条)**:`000` 与 `502` 是两码事——`502` 是服务端网关死、`000` 才是客户端到 ngrok 不通。确认公网不是 502 后:客户端进程连 `*.ngrok.app` 必须走本机代理(如 `127.0.0.1:7897`),但 MCP client 没继承代理环境 → `000`/not connected(带代理测则秒变 `401`)。修复:启动客户端前设 `HTTPS_PROXY=http://127.0.0.1:7897`(或在该 MCP server 配置里加代理),再重连 MCP 面板。**「之前能用现在断」两种可能:① 服务端网关挂了(公网 502);② 客户端进程这次没带代理环境(公网 200/401 但客户端 000)。用上面的一步 curl 先分清。** 注:ngrok 是海外中转隧道,大陆客户端连 ngrok 本身需代理是其固有特性,想彻底免代理需换国内直连/已备案域名反代,非服务端可消除。 |
| 401 | `API_KEY` 不匹配,核对客户端 header 与服务器 `.env`(注意本文档示例用 `<API_KEY>` 占位,须替换成开头的真实值) |
| `agent_run` 返回 failed | 看 error:引擎未登录(服务器 `claude`/`codex login`)、prompt 空、engine 不在白名单、`session_id` 非法(须为 UUID) |
| 续接没接上历史(答不出第3步的 42) | 多半传错字段:接力靠 `session_id`,**不是** `task_id`;或服务器侧该会话历史已被 CLI 清理(走 failed),重开首轮即可 |
| `ifind_*` 取数鉴权错 | 服务器 `IFIND_MCP_JWT` 过期,重取新串改 `.env` 重启网关 |
| `mx_ds_*` 工具列空或取数报错 | `MIAO_XIANG_MCP_KEY` 缺失/失效,改 `.env` 重启网关;或上游 API 授权未开通 |
| `tushare_ds_*` 工具列空或取数报错 | `TUSHARE_MCP_TOKEN` 缺失/失效,改 `.env` 重启网关;或 token 积分不足该接口权限 |
| **只剩 4 个 `agent_*` 工具**(全上游列空) | 上游 TLS `CERTIFICATE_VERIFY_FAILED`:根证书只在 OS 库、不在 certifi。确认 `.venv-mcp` 装了 `truststore` 且 `server.py` 启动时 `inject_into_ssl()`,重启网关 |
| 公网 mcp 偶发 000/404(服务端自测时) | 在服务器上测才算瞬态:隧道刚重建,等几秒重试;持续则查 `ngrok.log` 有无 `already online`(多会话冲突)。**客户端侧的 `000` 优先按上面「需走代理」排查,别当瞬态** |

## 六、安全与注意事项(两通道共用)

- **安全**:服务暴露在公网,**唯一防线是 `API_KEY`**。务必保管好 key,不要发到公开渠道;怀疑泄露立即在服务器 `.env` 改 `API_KEY` 并重启 `StockDataService` + MCP 网关。通道 ② 的 agent 工具尤其危险——拿到 key 即可在服务器无确认执行任意命令,**key 只发可信机器**。
- **单点登录**:iFinD 与 Wind 均为单点登录,账号同一时间只能一处登录,本服务各占用一个登录位;别处再登会互相挤掉。Wind / TDX 还依赖本机对应桌面终端已开启并登录。四个 SDK 共用同一把全局锁串行化取数,吞吐不随并发线性提升(设计预期)。
- **限频/流量**:EM 截面/序列每分钟 ≤700 次,并按品种有流量额度;iFinD 为单点登录、且服务内部用全局锁串行化所有请求,高并发会排队变慢。
- **Wind 取数 + 受控交易**:`/wind/*` 取数只读免开关;交易/调仓类函数(`torder`/`tlogon`/`tcancel`/`wupf` 等)经 `/wind/call` 调用,默认被总开关 `WIND_TRADING_ENABLED=false` 拦截(`503`),开启后放行并写审计(Wind 无固定 order 端点,函数签名各异,不做 dry-run/护栏,靠 API Key + 总开关兜底,需交易权限的 Wind 终端)。`wsq` 实时为异步回调,HTTP 同步端点不暴露。
- **TDX 取数 + 独立交易腿**:`/tdx/*` 取数只读;**交易能力已放开**(与 QMT `/qmt/*` 平级、两个独立券商接口)。写端点 `POST /tdx/order`(下单)、`POST /tdx/cancel`(撤单)默认被 `TDX_TRADING_ENABLED=false` 拦截(`503`),开启后仍**默认 dry-run**(`confirm=true` 才真发),并过四道护栏(金额/白名单/当日笔数/交易时段)+ 全量审计。读端点 `GET /tdx/asset|positions|orders|health` 不受开关限制。🔴 撤单需 `stock_code + order_id`;下单参数 `side=buy|sell`、`price_type=limit|market`。通用透传 `/tdx/call/{method}` 仍硬拦 403(交易/账户查询/写客户端/会话类)。证券代码须为标准格式(如 `600519.SH`),日期格式 `YYYYMMDD`。
- **公网放开真实交易(QMT/TDX/Wind 总开关)前需与服务方显式确认。**
