# StockSDK 运维手册

> 面向「不看代码也能照做」的运维。覆盖启停、健康探活、会话自愈、冷启动首登失败、
> EM 令牌、ngrok、压测、回滚。配套 [`CLAUDE.md`](../CLAUDE.md)（架构全貌）。

## 0. 凭据与安全底线

- 凭据只在本机:REST 服务的 iFinD 走 `.env` 的 `THS_USER/THS_PWD`;EM 走 `userInfo` 令牌
  （或 `.env` 的 `EM_USER/EM_PWD`）;MCP 网关走 `.env` 的 `IFIND_MCP_JWT`。**三套独立**。
- `.env`、`userInfo`、`*.log` 已 gitignore。任何 commit/push 前确认 diff 无凭据。
- 公网（ngrok）暴露时 `API_KEY` 必配,否则取数接口不鉴权。

## 1. 启停（都是看门狗，崩溃 5s 自动重启）

| 动作 | 命令 |
|---|---|
| 起 REST(:8000) | `start_server.bat` |
| 起 MCP 网关(:8765) | `start_mcp_gateway.bat` |
| 起公网隧道 | `start_ngrok_gateway.bat`（一个 ngrok agent 同时暴露 8000+8765） |

> 机器上还注册了计划任务 **StockDataService**（每分钟兜底自愈）:kill 掉 8000 进程后约 1 分钟内会被重新拉起。
> REST 必须**单 worker**（看门狗已按此拼命令）,不要手动加 `--workers`。

## 2. 健康探活

```bash
# 浅:进程是否存活(不持锁、不鉴权)
curl -s http://127.0.0.1:8000/health
# → {"status":"ok"}

# 深:实测 iFinD + EM 各取一笔最小数据 + 时延(需 API_KEY)
curl -s -H "X-API-Key: <key>" http://127.0.0.1:8000/health/deep
# → {"status":"ok|degraded","ifind":{"status":"ok","latency_ms":..},"em":{...},"as_of":..}
```

- `/health/deep` 任一源故障**不返回 5xx**,整体仍 200,故障在该源 `status=down` + `reason`（已脱敏）里体现。
- 看到 `degraded` → 看是哪个源 down,按下面对应小节处置。

## 3. 会话自愈（多数情况自动，无需人工）

两个 SDK 单会话、单点登录,会话会被悄悄踢掉（同账号别处登录、iFinD 单点冲突、令牌超时）。
`sessions.em_exec/ths_exec` 在取数命中失效码时**自动强制重登并重试一次**:EM 用 `ForceLogin=1` 抢登,
iFinD 先 `Logout` 再 `Login`。表现为偶发一次慢、随后恢复,**通常无需手动重启**。

> 预防:服务用的账号别同时在桌面客户端或其他脚本登录同一账号。

## 4. iFinD 冷启动首登失败（自愈覆盖不到，需人工）

**症状**:`/ths/*` 返回 `502 {"detail":"iFinD 登录失败:-1"}`。`-1` 是 `THS_iFinDLogin` 通用登录失败码。
**根因**:`.env` 的 `THS_USER/THS_PWD` 过期或换号后未更新——服务用旧凭据**冷登**失败。
自愈只处理「已登录后被挤掉」的重登,冷启动首登就 -1 时重登仍撞同一个 -1,所以要人工。

**处置**:
1. 改 `.env` 的 `THS_USER/THS_PWD` 为新凭据（EM 侧不受影响,凭据独立）。
2. 找 uvicorn 子进程 PID:`netstat -ano | grep ":8000" | grep LISTENING`。
3. `taskkill //PID <PID> //F`——看门狗 5s 后用新 `.env` 冷登重启,**无需手动起服务**。
4. 轮询 `/health` 到 200,再测:
   `/ths/basic?codes=300033.SZ&indicators=ths_stock_short_name_stock&params=300033.SZ`
   成功返回 `"同花顺"`、`errorcode:0`。

## 5. EM / Choice 令牌与激活

- 默认用本机 `userInfo` 令牌登录。令牌失效 → 重新激活:
  运行 `D:\dev\Project\EMQuantAPI_Python\python3\libs\windows\LoginActivator.exe`,
  输入绑定手机号 + 验证码,生成新 `userInfo`。
- 想用账密登录（不生成令牌）:在 `.env` 加 `EM_USER/EM_PWD`。
- 运行依赖:本机已装 Microsoft Visual C++ 2010 可再发行组件包（x64）。
- 验证:`.venv-em\Scripts\python verify_em.py`。

## 6. ngrok 重连

- 配置 `ngrok-gateway.yml`（authtoken 不在此,走全局 `%LOCALAPPDATA%\ngrok\ngrok.yml`）。
- 免费版同一 authtoken 仅 1 个 agent 会话,故两个 tunnel（api+mcp）合在一个配置里。
- mcp tunnel 若无第二个固定域名会分到随机域名,看 `ngrok.log` 或 ngrok 后台。
- 断了重起:`start_ngrok_gateway.bat`。

## 6b. frp 香港隧道（ngrok 平替,大陆可达,自建自控）

> 动机:ngrok 大陆链路不稳。改为本机 `frpc` 主动连出到**自建香港 frps 节点**转发。与 ngrok 并行独立,可同时跑,切换稳定后再停 ngrok。记忆见 `frp-hk-tunnel-migration`,计划见 `.claude/plans/frp-hk-tunnel-migration.plan.md`。

**架构**:本机 frpc →（TLS 隧道）→ 香港 frps `47.76.104.225:7000` → 转发两条内部口 → **Caddy 终结 HTTPS 对外(域名 + Let's Encrypt)**:
- `api`:公网 `https://api.jiantx.net`（443）→ Caddy → frps `127.0.0.1:18000` → 本机 `8000`（REST + WS）
- `mcp`:公网 `https://mcp.jiantx.net/mcp`（443,SNI 路由)→ Caddy(`flush_interval -1` 保 SSE)→ frps `127.0.0.1:18765` → 本机 `8765`
- 明文口 `18000`/`18765` 被主机防火墙 nftables **只对 loopback 放行**,公网访问被 drop → **只允许 HTTPS**。
- 🔴 旧的 IP 直连(`https://47.76.104.225` / `:8443`)与自签证书**已停用**,只认域名(SNI 不匹配的 IP 裸连不再服务)。

**本机启停**（镜像 ngrok 那套看门狗 + 计划任务）:
- 起:`schtasks /Run /TN FrpcTunnel`(开机自启 + PT1M 自愈 + IgnoreNew 单实例)
- 停:`schtasks /End /TN FrpcTunnel`
- 配置 `frpc.toml`（token 走 `{{ .Envs.FRP_TOKEN }}` 环境变量插值）+ `frpc.secret.bat`（**不入库**,设 FRP_TOKEN）+ `start_frpc.bat`（看门狗）。二进制 `frp/frpc.exe`（不入库）。
- 重装计划任务:`powershell -ExecutionPolicy Bypass -File register_frpc_task.ps1`

**香港节点**（Debian 12,用户 `admin` sudo 免密,2C2G,SSH 私钥 `.ssh-frp/hk_frps`）:
- 起停:`sudo systemctl start|stop|status frps`(Restart=always 自愈)
- 配置 `/etc/frp/frps.toml`(bindPort 7000、`transport.tls.force`、token、dashboard 127.0.0.1:7500)
- 看隧道在线:`curl -s -u admin:<pw> http://127.0.0.1:7500/api/proxy/tcp`(dashboard 仅本机/SSH 内)
- frps 日志:`/var/log/frps.log`

**探活**（带 key,经香港公网,验证端到端;Let's Encrypt 受信证书,**无需 `-k`**）:
```bash
curl -H "X-API-Key: <key>" https://api.jiantx.net/health/deep          # REST + 真实取数
curl -H "X-API-Key: <key>" "https://api.jiantx.net/em/css?codes=600519.SH&indicators=CLOSE"
curl -H "X-API-Key: <key>" -H "Accept: application/json, text/event-stream" \
  -X POST -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"t","version":"1"}}}' \
  https://mcp.jiantx.net/mcp                                            # MCP:返回 SSE event 即通
```

**HTTPS 终结（Caddy,香港节点)**:
- Caddy 二进制 `/usr/local/bin/caddy`,配置 `/etc/caddy/Caddyfile`,systemd `caddy.service`（Restart=always）。
- 证书 **Let's Encrypt 受信任**(`api.jiantx.net`/`mcp.jiantx.net`),Caddy 自动申请 + **自动续期**(90 天,到期前自动换),存 `/var/lib/caddy` ACME 数据目录。
- 签发走 **TLS-ALPN-01 挑战(443 口)**,**无需开 80 口/改安全组**;两个主机名都在 443,靠 **SNI 路由**(有真实域名,不再需要 IP 期那种 `:443`/`:8443` 裸端口拆分)。
- 旧自签 `/etc/caddy/ip.crt`+`ip.key` 留作回滚,备份配置 `/etc/caddy/Caddyfile.selfsign.bak`。
- 起停:`sudo systemctl start|stop|restart|status caddy`;看证书:`echo | openssl s_client -connect api.jiantx.net:443 -servername api.jiantx.net 2>/dev/null | openssl x509 -noout -issuer -dates`。

**只允许 HTTPS（nftables,香港节点)**:
- 表 `inet frpguard`：`iif lo accept` + `tcp dport {18000,18765} drop`（放行 Caddy 走 loopback,drop 公网明文）。
- 持久化在 `/etc/nftables.conf`,`nftables.service` 开机自启。
- 查规则:`sudo nft list table inet frpguard`。

**🔴 接入踩坑（已解,重装勿重走）**:
1. **本机直连 GitHub 被重置**（大陆网络）→ 香港机下 win zip 再 `scp` 回;香港无 unzip 用 `python3 zipfile` 解。
2. **frp `includes` 不合并 `auth.token` 等全局字段**（只合并 proxies/visitors）→ 用 `{{ .Envs.FRP_TOKEN }}` 环境变量插值。
3. **`.bat` 必须 CRLF + secret.bat 纯 ASCII 注释**:LF 行尾/中文注释在 GBK cmd 下解析乱致 frpc 秒退(frpc.log 无新条目=配置加载前退)。
4. `register_frpc_task.ps1` 的 `$PSScriptRoot` 某调用下为空→已加 `$MyInvocation` 兜底。

**✅ 安全现状**:已做 HTTPS-only(Caddy **Let's Encrypt 受信证书** + nftables 封明文),`X-API-Key` 全程加密。域名 + 受信证书后客户端**零配置、去掉 `verify=False`**,既加密又防 MITM,可用于真实交易公网流量。

## 6c. 多机 Agent 协同（hub + 对等机）

- **形态**：本机（hub）网关 `MCP_GATEWAY_MODE=full`（默认）；对等机跑同一仓库、
  `MCP_GATEWAY_MODE=agent-only`（只暴露 agent 工具），经 frp 各占一个公网端口
  （约定 pc2=18766 / pc3=18767 / mac1=18768；proxy name 必须带机器名后缀防与 hub 的 `mcp` 撞名）。
  对等机支持 **Windows 与 macOS**（mac 用 launchd KeepAlive 替代计划任务+看门狗，
  差异全部收口在部署手册的系统分支里）。
- **hub 接入**：`.env` 填 `MCP_PEERS=pc2=http://47.76.104.225:18766/mcp,…` +
  `PEER_PC2_API_KEY=…`（每机独立 key），重启网关后日志见 `已挂载对等机 pc2 -> 前缀 peer_pc2`。
- **对等机全量部署/排错**：见 [`docs/部署-对等机AgentMCP.md`](部署-对等机AgentMCP.md)
  （可直接丢给对等机 Claude Code 自动执行）。对等机计划任务：`MCPGatewayBoot`
  （register_mcp_gateway_task.ps1）+ `FrpcTunnel`（register_frpc_peer_task.ps1）。
- **连通自检**：`set CHECK_API_KEY=<key>` 后
  `.venv-mcp\Scripts\python scripts\check_mcp_gateway.py [url]`（缺省本机 8765）。
- **peer 宕机**：hub 侧 5s connect 超时兜底，其余工具不受影响；长期关机的 peer 从
  `MCP_PEERS` 摘除。session_id / agent_memory 均不跨机。

## 7. 压测（loadtest/，验证高可用证据，不在生产路径）

⚠ **运行约束（踩过的坑）**:
- 必须用 `.venv-api\Scripts\python.exe` 跑。
- **别用 Git-Bash 直接跑**:MSYS 会把 `--target /health` 转义成 Windows 路径,导致全部请求失败
  却报出 ~100k rps 的假结果。用 `cmd`,或设 `MSYS_NO_PATHCONV=1`。
- 跑测试套件要 `PYTHONPATH=.` 才能 import stocksdk。

**已验证基线（本机,供对照）**:
- `/health` 并发 1→8:吞吐 259→502 rps;8→32 趋平 ~500 rps,p50 16→63ms。
  单 worker 事件循环饱和、取数全局锁串行,**吞吐不随并发翻倍是设计预期,非缺陷**。
- 崩溃自愈:看门狗层 kill 后约 6.1s 恢复;计划任务兜底约 1 分钟内拉起。

详见 `loadtest/README.md`。

## 8. 回滚原则

- 本仓库是只读数据提供方,**没有交易/下单可回滚**。
- 出问题先看 `/health/deep` 定位故障源,再对照本手册第 4/5 节处置单源。
- 改代码导致的回归:`git revert` 对应 commit,看门狗自动用新代码重启。
- 改完务必跑 `.venv-api\Scripts\python -m pytest --cov=stocksdk` 再发布。

## 9. 常见症状 → 处置速查

| 症状 | 大概率根因 | 去哪 |
|---|---|---|
| `/ths/*` 502 登录失败:-1 | iFinD 冷登凭据过期 | §4 |
| `/ths/basic` 返回 null 但 errorcode:0 | 指标码/函数用错或无数据权限 | 见 [`docs/数据源对照-StockSDK提供.md`] |
| `/health/deep` 某源 down | 会话被挤/令牌失效 | §3 自愈;不恢复看 §4/§5 |
| EM 取数报未登录 | userInfo 令牌失效 | §5 重新激活 |
| 远端 MCP 拿不到工具 | IFIND_MCP_JWT 过期 | 改 `.env` 重启网关,见 `docs/MCP中转网关.md` |
| 压测报 ~100k rps 假结果 | Git-Bash 路径转义 | §7 用 cmd |
