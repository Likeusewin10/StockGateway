# GUI 自动化可行性方案

> 目的：评估在 StockSDK 现有 SDK/HTTP 服务之外，增加一条“桌面 GUI 人机协同/RPA”路径的可行性。
>
> 边界：本方案不用于绕过厂商风控、验证码、反自动化、权限限制或服务条款；仅用于在已登录、已授权的本机桌面客户端内，做低频、可审计、可人工确认的操作。涉及真实交易时，默认只允许“查询/预填单/截图确认”，最终提交必须人工确认，除非后续另行明确授权并加入护栏。

## 结论

可以做，但不建议把 GUI 自动化当作主取数通道。

更适合的定位是：

1. **补洞路径**：SDK/API 被限流、单点冲突、某些功能 SDK 没开放时，用 GUI 导出或查询补数据。
2. **运维辅助**：自动检查终端是否登录、弹窗是否卡住、API 插件是否正常、窗口是否在线。
3. **半自动交易辅助**：打开交易界面、填入证券/价格/数量、截图给人确认；默认不点击最后“买入/卖出/确认”。
4. **字段/报表导出**：用客户端自带“导出 Excel/CSV”功能，低频导出报表，比 SDK 枚举/暴力探测更稳。

不适合：

- 高频取数、实时行情主链路。
- 绕过账号权限、验证码、风控弹窗、反爬/反自动化检测。
- 无人值守真实下单。
- 大规模逐项点击遍历字段树。

## 为什么 GUI 不一定更安全

GUI 操作看起来像人，但如果做成高频、无间隔、批量遍历、长期无人值守，同样可能触发风控；而且比 SDK 更脆弱：

- 窗口标题、控件层级、按钮文案、坐标都会随版本变化。
- 弹窗、登录超时、网络波动会打断流程。
- GUI 只能单桌面会话运行，受锁屏、远程桌面断开、分辨率/DPI 影响。
- 错点按钮的风险更高，尤其交易界面。

因此建议把 GUI 层做成“慢、少、可审计、可截图回放、失败即停”。

## 推荐架构

在现有 StockSDK 旁边新增一个独立模块：`stocksdk/gui/`。

```text
stocksdk/gui/
  __init__.py
  backends.py          # pywinauto/uiautomation 后端封装
  apps.py              # 终端进程/窗口发现：iFinD、Choice、Wind、TDX、QMT
  health.py            # 登录状态、弹窗、窗口在线检测
  export.py            # 导出 Excel/CSV 的半自动流程
  trading_assist.py    # 交易界面预填，默认不最终提交
  audit.py             # 操作日志 + 截图路径
scripts/gui_probe.py   # 探测当前桌面窗口和控件树
```

再新增一组 HTTP 端点时，建议单独命名：

```text
GET  /gui/health
GET  /gui/windows
POST /gui/export/{vendor}
POST /gui/trade/prefill/{vendor}
POST /gui/screenshot/{vendor}
```

并独立开关：

```env
GUI_AUTOMATION_ENABLED=false
GUI_TRADING_ASSIST_ENABLED=false
GUI_ALLOW_FINAL_SUBMIT=false
GUI_AUDIT_DIR=audit/gui
```

默认全部关闭。开启后仍要求 API Key，并写审计。

## 技术选型

Windows 桌面优先选：

1. **pywinauto**：成熟，支持 Win32/UIA 两种后端，适合 Windows 原生客户端。
2. **uiautomation**：控件树读取更直接，有时对国产金融客户端更好用。
3. **截图 + OCR/图像定位**：仅作为兜底，避免纯坐标点击。

不建议优先用纯坐标脚本，因为 DPI/窗口位置一变就失效。

## GUI 操作安全分层

### 查询/导出类

允许自动执行，但必须：

- 限频。
- 每步等待控件稳定。
- 失败即停，不盲点。
- 保存审计：时间、应用、窗口、操作、输出文件、截图。

### 交易类

默认只做“辅助”：

1. 打开交易窗口。
2. 填入证券代码、方向、价格、数量。
3. 截图并返回给调用方。
4. 等人确认。

默认不点击最终“确认/提交/买入/卖出”。

如果未来确实要无人值守最终提交，必须另加：

- 独立总开关 `GUI_ALLOW_FINAL_SUBMIT=true`。
- 和 QMT/TDX 一样的金额、白名单、当日笔数、交易时段护栏。
- 强审计截图：提交前、提交后、委托回报。
- 小额灰度验证。

## 最小可行路径（MVP）

### Step 1：窗口探测

写 `scripts/gui_probe.py`：

- 列出当前桌面窗口标题、进程名、句柄。
- 对指定窗口 dump 控件树。
- 保存截图。

目标是确认 iFinD/Choice/Wind/TDX/QMT 哪些窗口可被 UIA 识别。

### Step 2：只读健康检查

做 `/gui/health`：

- 进程是否存在。
- 主窗口是否存在。
- 是否有登录/超时/验证码/错误弹窗。
- 当前窗口截图路径。

### Step 3：导出固定报表

先选一个低风险场景：例如 iFinD/Choice 的某个固定报表导出。

要求：

- 不遍历全量树。
- 只操作明确菜单/按钮。
- 输出到 `downloads/gui_exports/`。
- 生成审计 JSON。

### Step 4：交易预填，不提交

如果目标是交易客户端：

- 实现 `prefill_order`。
- 填好后截图。
- 返回“等待人工确认”。

## 与现有 SDK 服务的关系

现有 REST/SDK 层仍应保留为主路径：

- EM/iFinD/Wind/TDX 数据接口已有封装。
- QMT/TDX 交易接口已有护栏和审计。
- GUI 层只是补洞和人工协同，不替代主链路。

推荐调用优先级：

```text
官方/已授权 API 或 SDK
  -> 现有 StockSDK HTTP 端点
  -> GUI 导出/人工协同补洞
  -> 放弃，不做风控绕过
```

## 当前项目落点

本仓库已经有：

- 多源 SDK 服务封装。
- API Key 鉴权。
- 交易开关/dry-run/护栏/审计思路。
- 文档里已有“超级命令/数据浏览器导出指引”。

所以 GUI 自动化最自然的切入点不是重写所有端点，而是把已有人工导出指引自动化成可控流程：

1. 先做窗口/控件探测。
2. 再做导出类自动化。
3. 最后才考虑交易预填。

## 风险提示

如果出发点是“SDK 容易风控，所以用 GUI 绕过去”，这条路不稳，也不建议。更稳的做法是降低调用频率、使用明确授权的接口、按字段手册精准取数、避免遍历/暴力探测，并把 GUI 作为低频人工协同工具。
